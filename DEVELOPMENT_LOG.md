# NotchFix 开发日志

## 项目目标
解决 macOS M4 刘海屏导致菜单栏图标被隐藏的问题。

## 尝试的方案

### 方案 1: Accessibility API 直接隐藏图标 ❌
**时间**: 2026-03-13 初期

**实现**:
- 使用 `AXUIElementCreateSystemWide()` 扫描菜单栏
- 尝试通过 `kAXHiddenAttribute` 隐藏图标
- 使用 `IconManager` 管理隐藏状态

**遇到的问题**:
1. **权限问题**: 错误码 `-25205` (kAXErrorAPIDisabled)
   - 即使在系统设置中授予了辅助功能权限，仍然报错
   - 原因：开发版本 (`.build/debug/NotchFix`) 和安装版本 (`/Applications/NotchFix.app`) 签名不同
   - macOS 对每个二进制文件单独管理权限

2. **编译环境问题**: 
   - Xcode Command Line Tools 损坏 (`xcrun` 报错)
   - 无法正常编译 release 版本
   - 影响应用签名和权限识别

3. **API 限制**:
   - `kAXHiddenAttribute` 不是所有应用都支持
   - 即使有权限，也无法保证能隐藏所有图标

**结论**: 技术上可行，但用户体验差（权限配置复杂，兼容性差）

---

### 方案 2: NSWorkspace 应用快速启动器 ⚠️
**时间**: 2026-03-13 中期

**实现**:
- 使用 `NSWorkspace.shared.runningApplications` 扫描应用
- 通过 `activationPolicy` 判断是否为菜单栏应用
- 提供应用列表，点击激活应用

**优点**:
- ✅ 无需辅助功能权限
- ✅ 实现简单，技术稳定
- ✅ 提供额外的快速启动功能

**问题**:
- ❌ 无法准确识别哪些应用真的在菜单栏有图标
- ❌ 无法隐藏原图标
- ❌ 不符合用户需求（用户要的是"隐藏图标"，不是"应用启动器"）

**结论**: 功能偏离需求，用户不满意

## 技术难点总结

### 1. macOS 权限系统
- 辅助功能权限是**按二进制文件**授予的，不是按应用名
- 每次重新编译，签名变化，需要重新授权
- 解决方案：安装到 `/Applications` 后签名稳定

### 2. Accessibility API 的局限性
- 不是所有应用都支持 `kAXHiddenAttribute`
- 需要用户手动授权，无法自动化
- 权限提示对用户不友好

### 3. 编译环境问题
- Xcode Command Line Tools 损坏会导致无法编译
- `xcrun` 错误影响 Swift Package Manager
- 需要用户手动修复：`sudo xcode-select --reset`

## 经验教训

1. **权限问题要早发现早解决**
   - 应该在项目初期就测试权限流程
   - 提供清晰的权限配置指南

2. **技术选型要考虑用户体验**
   - 不能只考虑技术可行性
   - 要考虑用户是否愿意授予权限

3. **开发环境很重要**
   - Xcode Command Line Tools 的问题影响了整个开发流程
   - 应该提供环境检查脚本

4. **需求要明确**
   - 用户要的是"隐藏图标"，不是"应用启动器"
   - 方案 2 虽然技术上可行，但偏离了需求
